# PR #494 Review — opus-4-7

## Verdict
SHIP-WITH-FIXES — demo flow works, but there are two real correctness/UX bugs that can bite mid-demo, and a couple of security holes that don't matter for demo day but should be ticketed for post-hackathon. The biggest demo risk is the known 0G mainnet `FLOW_CONTRACT` blocker called out in the PR description; that's an operator issue, not a code bug.

## HIGH

| # | File:line | Issue | Fix |
|---|---|---|---|
| H1 | `apps/web/src/pages/OwnerEditor.tsx:106-130` | Real-mode `loadToken` catch only logs to `txLog` and **does not clear `data`/`owner` on failure**. If a judge types a token id that isn't minted yet, the on-chain call reverts (`TokenNotFound`) but the UI keeps showing the *previous* token's owner/data. State is silently stale → screenshots will mislead. | In the catch, also `setOwner('demo-owner')` and `setData(demoData(tokenId, notes))` (or at least clear them) so the displayed state always matches the load attempt. Optionally surface the error inline (red banner) rather than only in the log. |
| H2 | `apps/web/src/pages/OwnerEditor.tsx:160-176` | `updateMetadata` writes to localStorage with the **old `data` value** before the on-chain call returns. Sequence is `setData(nextData)` → `persistDemoState(owner, nextData)` → `writeContract`. If the wallet user rejects the tx OR the tx reverts, the `clanworld:inft-demo:<tokenId>` row in localStorage is already updated to `nextData` and the cockpit `ZeroGTab` will display the **post-update state even though the chain didn't change**. Demo lies. | Move the `persistDemoState` + `setData` calls into the success branch (after `await client.writeContract` resolves and `loadToken` re-fetches), or wrap the optimistic update in a try/catch and revert on failure. |
| H3 | `packages/contracts/src/ClanAgentNFT.sol:204-214` | `_transfer` does **not** clear `usageAuthorizations[tokenId][*]` or `delegatedAccess[tokenId]` on transfer. After Alice transfers to Bob, anyone Alice previously authorized via `authorizeUsage` can still call `updateMetadata` on Bob's token (the access control checks `usageAuthorizations[tokenId][msg.sender]`). This violates the ERC-7857 ownership-handoff invariant. | Either (a) namespace authorizations by owner address — `usageAuthorizations[tokenId][owner][user]` — or (b) bump a per-token `authorizationsEpoch` counter on transfer and gate `updateMetadata` on `authorizationsEpoch[tokenId] == storedEpoch[user]`. Pure-mapping fix is iteration-over-users which isn't viable on-chain. For demo: skip; for production: must-fix. |
| H4 | `apps/server/convex/inft.ts:46-66` | `mirrorToken`/`mirrorTransfer`/`mirrorMemoryEntry`/`mirrorBulletin` are **public Convex mutations with no auth gating**. Anyone with the deployment URL (publicly discoverable from the bundle) can spam fake transfer rows or overwrite the displayed `owner`/`dataHash`. During the live demo a malicious viewer could grief the cockpit display. | Gate behind an `internalMutation` + a Convex action that validates the chain event via on-chain read or signature, OR add a shared secret HMAC header that the indexer attaches. For the demo specifically, at minimum verify the row matches what `getInftDemoState` would compute against the chain (e.g. server-side check `owner === ownerOf(tokenId)` via `INDEXER_RPC`). |

## Notes

**Demo-day blocker risk** — The PR description says "FLOW_CONTRACT submit reverts on mainnet despite correct address — operator action open." That's the single biggest risk to the demo. The smoke test (`infra/0g/smoke-test.ts`) is the gate that proves whether the 0G memory write actually works. Run it once, end-to-end, on the demo machine, before showtime. If it fails, the iNFT story degrades to "Base + local memory + cockpit-only" which the cockpit handles gracefully (file fallback) but the judges will lose the 0G headline.

**ABI ↔ Solidity alignment** — Spot-checked `IInftClient.ts`'s `CLAN_AGENT_NFT_ABI` against `ClanAgentNFT.sol`. `ownerOf`, `currentDataHash`, `encryptedKeyHash`, `intelligentDataOf`, `updateMetadata`, `iTransfer`, `authorizeUsage`, `revokeAuthorization` all match argument types and orderings. ✓

**Hashing alignment** — `hashIntelligentData` (TS, viem) uses `parseAbiParameters('bytes32 rolling, string label, bytes32 dataHash, string uri')` with `keccak256` — matches Solidity `keccak256(abi.encode(rolling, label, dataHash, uri))`. ✓ (Test in `inftClient.test.ts` confirms.)

**`elderToClanId` type** — Verified in `packages/runner/src/types.ts` it's `Record<ElderId, string>`, so passing it as `clanId: string` to `createMemoryStore` typechecks. ✓

**Convex schema duplication check** — `bulletins` and `memoryEntries` only defined once in `apps/server/convex/schema.ts`; no conflict elsewhere. ✓

**Reentrancy guard not in place** — Mentioned in M1. The contract is hackathon-safe with the bundled mock verifier, but the `setVerifier` admin function is open. If post-hackathon you ever ship to mainnet without addressing M1, that's a real attack surface.

**Test coverage gap** — `ClanAgentNFT.t.sol` covers happy path + non-owner revert + authorized-user update + empty-proof reject (✓). Doesn't cover: post-transfer authorization persistence (would catch H3), `updateMetadata` with unauthorized user, `delegateAccess` end-to-end. Acceptable for hackathon scope.

**Suggested merge action** — Land H1 + H2 (UI correctness, ~10 min total fix) before demo. Defer H3, H4, M1–M5 to a follow-up issue. The H1/H2 fixes are pure local edits to `OwnerEditor.tsx`; no contract redeploy needed.
