# Diamond contract review

## Architecture

The diamond split is now reasonable.

`Diamond.sol` is a tiny dispatcher:

* Constructor installs the `diamondCut` selector at `src/diamond/Diamond.sol:8-21`.
* Fallback loads the facet address from diamond storage and delegatecalls it at `src/diamond/Diamond.sol:26-38`.

`LibStorage` uses a namespaced app-storage slot. That is the right model for a diamond because all facets mutate the same world state through one storage root. It also means this diamond is **not storage-compatible with the old monolith**. If old state exists, migration must be snapshot/claim/restoration, not “point proxy at old storage.”

## Diamond-specific safety

The code is almost there but needs a final hardening pass.

### MUST: zero owner and empty selector protection

`LibDiamond.setContractOwner()` emits a transfer and sets the owner without checking zero at `src/diamond/lib/LibDiamond.sol:47-52`. `OwnershipFacet.transferOwnership()` checks zero, but the constructor path does not.

Also, empty selector arrays are accepted. That can pollute `facetAddresses` or create confusing loupe state. The typical invariant should be:

```solidity
require(selectors.length > 0, "Diamond: empty selector cut");
```

for add, replace, and remove.

### MUST: init calldata validation

This path currently silently ignores nonempty calldata:

```solidity
if (init == address(0)) {
    return;
}
```

at `src/diamond/lib/LibDiamond.sol:187-190`.

That should be:

```solidity
if (init == address(0)) {
    require(data.length == 0, "Diamond: init calldata without init");
    return;
}
```

Otherwise a bad deployment/cut can appear to include initialization but do nothing.

### SHOULD: explicit freeze mode

Owner-only `diamondCut` is fine for dev/testnet. For production, the upgrade surface is the most dangerous part of the diamond.

Recommended pattern:

1. Deploy mutable under multisig/timelock.
2. Run testnet/season zero.
3. Perform final cut.
4. Freeze `diamondCut` permanently or require a long timelock with public notice.

Since this is a game with deterministic state and GOLD-like balances, I would add a one-way `freezeDiamondCuts()`.

## Heartbeat and season behavior

The previous “finalize one tick too early” bug appears fixed. Current finalization requires `currentTick >= seasonEndTick` at `FinalizeSeasonFacet.sol:18`, and heartbeat processes the main tick normally until `currentTick` reaches the end.

Remaining issue: after season end but before finalization, `heartbeat()` succeeds as a silent no-op:

```solidity
if (isSeasonFinalizationPending(s)) {
    return;
}
```

at `src/diamond/lib/LibHeartbeat.sol:23-25`.

That is operationally bad. Keepers can repeatedly burn gas. Monitoring sees successful heartbeats but no tick movement. I would make this revert with `"ClanWorld: finalize season first"` or emit a very explicit skip event and rate-limit the next heartbeat.

## Settlement and cleanup

The diamond port now tracks more simulation-reservation state than the prior branch. `SettlementSimulation` includes reserved wood, iron, wheat, and blueprint counters, which addresses a previous major issue.

Death cleanup also improved. The storage bandit path now clears defender/reservation state and releases target-side defenders/bandit attacks when a clan dies. That is much closer to what the monolith did.

One invariant to test hard:

* Any path that deactivates an active mission must also clear side effects:

  * upgrade reservation,
  * defender registration,
  * scheduled market,
  * targeting/defending lists,
  * pending counts.

I saw the main kill paths handled, but the release gate should include invariant-style tests because this is exactly where diamond refactors lose internal coupling.

## Bandits

The bandit model looks materially better:

* Spawn-region allowlist is explicit.
* Target selection settles candidate regions first.
* Theft uses spendable balances instead of raw vault balances.
* Attack resolution has cleanup paths.

A remaining edge case: `LibBanditLifecycle.resolveBanditAttack()` falls back to resetting the bandit if the target clan is not fully settled. With the current heartbeat settling all clans first, this should usually not happen. But because settlement has a max backlog cap, a long-stale clan can still be under-settled. I would add a test that creates a >200-tick backlog and proves the bandit either eventually resolves or cannot grief the state machine forever.

## Views and spec drift

The view layer still has important drift.

`SnapshotViewsFacet.getWorldSnapshot()` says it returns leaderboard `lootValue` that is settled, but it uses raw clan storage:

```solidity
lootValue: LibScoring.lootValue(clan)
```

at `src/diamond/facets/SnapshotViewsFacet.sol:13-25`. The interface says `LeaderboardEntry.lootValue` is settled at `src/IClanWorld.sol:442-451`, and aggregator getters “MUST” follow derived getter semantics at `src/IClanWorld.sol:928-933`.

Other view drift:

* `BanditViewsFacet.getActiveBanditView()` still hardcodes `projectedTargetLootValue: 0` at `src/diamond/facets/BanditViewsFacet.sol:41-42`.
* `QuoteViewsFacet.getBanditTargetPreview()` returns stored target id, not a live preview, at `src/diamond/facets/QuoteViewsFacet.sol:8-10`.
* `RegionViewsFacet.getRegionPopulation()` uses raw `currentRegion`, so traveling workers can show stale placement at `src/diamond/facets/RegionViewsFacet.sol:8-45`.

This matters because the web app and Convex backend now depend on these views for live state and animation.

# Web app review, especially bandit animations

The web app is visually ambitious and has a lot of careful Pixi engineering. The bandit animation phase machine is a good direction: pure `computeBanditAnimPhase()`, explicit camp/telegraph/battle/no-battle/death phases, sprite asset loading, and separate live/demo rendering paths.

The correctness problem is that the live animation is still **not authoritative**. It infers too much from snapshots.

## MUST: live tick must not derive from logs

`WorldMap.tsx` currently derives live tick as:

```ts
Math.max(snapshot?.tick ?? 0, logs.length)
```

at `apps/web/src/WorldMap.tsx:1174-1180`.

That means Convex narration logs can advance animation time even if the chain snapshot has not advanced. This can cause bandit telegraphs, attack windows, day/night, and travel animations to desync from chain state.

In production:

* `snapshot.tick` is canonical.
* `agentLogs.length` is not canonical.
* If snapshot tick is stale, show stale/disconnected state.

Keep log-derived tick only behind `DEMO_MODE`.

## MUST: bandit combat animation should be event-driven

The current live bandit result is inferred by diffing previous and current snapshot:

* If the previous bandit disappears, it assumes `defeated` at `WorldMap.tsx:1241-1251`.
* If the bandit moves regions and had a target, it assumes `won` at `WorldMap.tsx:1256-1273`.
* If no target id existed, it assumes `no_battle_advance` at `1274-1279`.

The comments already admit the backend does not expose enough resolution data.

This will produce wrong animations for edge cases:

* bandit escapes,
* target died before attack,
* no valid target,
* defeat after filtering,
* backend misses one snapshot,
* delayed indexer update,
* bandit advances due to no target but stale target id remains.

The fix should be backend-first:

```ts
lastBanditResolution: {
  banditId,
  resolvedTick,
  fromRegion,
  toRegion?,
  targetClanId?,
  outcome: "defeated" | "won" | "escaped" | "no_target" | "no_battle_advance",
  defended?,
  txHash,
  logIndex,
}
```

Then the web phase machine consumes that exact event. Snapshot diff becomes fallback only.

## MUST: web currently ignores live chain combat events

This effect only runs in demo mode:

```ts
if (!DEMO_MODE || !rawChainEvents || rawChainEvents.length === 0) return;
```

at `apps/web/src/WorldMap.tsx:3831-3833`.

So even though Convex has real `chainEvents`, live mode does not use them for combat animation. That is backwards. Live mode should use indexed chain events; demo mode should use synthetic events.

## MUST: asset failure can hide live bandits

If no walk textures load, the phase machine returns without setting `assetsReady`:

```ts
if (neFiltered.length === 0 && seFiltered.length === 0) {
  return;
}
```

at `WorldMap.tsx:2615-2617`.

But in non-demo mode, the legacy red ring/static bandit marker is deliberately hidden at `WorldMap.tsx:3347-3364`.

Result: if the animation kit fails, production can render no bandit. You need a live fallback: static sprite + ring/countdown should appear when phase assets are unavailable.

## SHOULD: expose full bandit view to frontend

Frontend `SnapshotBandit` omits fields that the contract/backend already have or almost have:

* `attackAttemptsMade`
* `maxAttemptsRemaining`
* `projectedTargetLootValue`
* maybe `lastResolution`

Backend `getSnapshot.ts` drops these fields at `apps/server/convex/getSnapshot.ts:75-90`.

The frontend comments specifically say it cannot discriminate outcomes because attempts are not exposed. That is now a backend schema/API problem, not a Pixi problem.

## SHOULD: pause state must stop animation clocks

The contract snapshot includes `worldPaused` and `pausedAtTs` in `SnapshotViewsFacet.sol:31-45`. The Convex schema and `getSnapshot()` response do not expose those fields. The web app therefore cannot reliably stop tick-derived animation on pause.

Plumb pause state through backend and make map clocks respect it.

# Convex backend / indexer review

The backend is significantly more real than before.

## Good

* Fake heartbeat and real indexer are mutually exclusive at `apps/server/convex/crons.ts:6-13`.
* Fake heartbeat is gated by `CLANWORLD_USE_FAKE_HEARTBEAT=true` at `crons.ts:15-17`.
* Real indexer snapshot refresh/log polling are gated by `CLANWORLD_USE_REAL_INDEXER=true` at `crons.ts:19-22`.
* Heartbeat webhook validates bearer secret at `heartbeat.ts:131-138`.
* In real-indexer mode, webhook waits for receipt, validates `receipt.to`, validates engine mismatch, parses only engine logs, ingests them, and refreshes snapshot at `heartbeat.ts:151-238`.

That is a big improvement.

## MUST: expose contract snapshot fields

The contract has richer world state than the backend returns. `getSnapshot()` currently returns:

```ts
tick, tickEpoch, regions, clans, activeBanditId, bandit, derived season state
```

at `apps/server/convex/getSnapshot.ts:91-102`.

It does not return:

* `worldPaused`
* `pausedAtTs`
* `seasonFinalized`
* `nextHeartbeatAtTick`
* `currentSeasonNumber`
* `currentTickSeed`
* full leaderboard
* full bandit attempts/projected loot

Schema also lacks pause fields at `apps/server/convex/schema.ts:5-64`.

This backend drift directly harms web animation and HUD correctness.

## SHOULD: do not rederive season if contract snapshot already stored it

`getSnapshot.ts` derives season/winter locally from generated constants at `getSnapshot.ts:23-48`. That is okay as a fallback, but once the indexer stores `WorldSnapshot`, the API should return contract-provided fields. Otherwise a future contract constant/rule change can drift from the backend.

## SHOULD: add a compact event projection for animations

Rather than asking the web to parse raw event rows forever, Convex should project the latest meaningful animation events:

* `lastBanditResolution`
* `lastTickAdvanced`
* `lastWorldEvent`
* maybe `lastMarketExecution`

This keeps Pixi logic simple and prevents heuristic animation.

# Native Android app review

There are two mobile apps in the repo:

1. `apps/clan-world-mobile`: native Kotlin/Compose app.
2. `apps/mobile`: Expo/RN Seeker app.

The native Kotlin app is more polished than I expected. It has good theming, EncryptedSharedPreferences for session state, build-time URL hard-fails, and real Mobile Wallet Adapter integration. But owner authentication and WebView hardening are not production-grade yet.

## Good

* Build config refuses missing map/terminal URLs at `apps/clan-world-mobile/app/build.gradle.kts:24-37`.
* Session storage uses `EncryptedSharedPreferences` with AndroidKeyStore-backed master key at `SessionStore.kt:30-48`.
* Manifest has `allowBackup="false"` at `AndroidManifest.xml:15-18`.
* MWA package visibility is declared at `AndroidManifest.xml:4-13`.
* WebView lifecycle calls `destroy()` on release for the map view at `MapWebView.kt:45`.

## MUST: owner sign-in does not verify signatures

`owner/Mwa.kt` explicitly says:

> we do not verify the signature — any successful return is treated as signed in

at `apps/clan-world-mobile/.../owner/Mwa.kt:12-16` and again at `59-60`.

That is fine for a demo owner screen, but not for any clan control surface. A malicious or unrelated wallet could sign any message and the app would treat it as owner.

Minimum fix:

1. Challenge includes nonce, timestamp, clan id, app origin, and purpose.
2. Server verifies signature.
3. Server checks pubkey is linked to clan ownership / authorized operator.
4. Server returns short-lived session.
5. App never treats local wallet success alone as ownership.

## MUST: WebView origin allowlist

Native map WebView:

```kotlin
webViewClient = WebViewClient()
javaScriptEnabled = true
domStorageEnabled = true
loadUrl(BuildConfig.MAP_URL)
```

at `MapWebView.kt:36-42`.

Terminal WebView also enables JS/DOM and uses a custom client without origin allowlisting at `TerminalTab.kt:121-168`.

Because JS is enabled, the app should restrict navigation to known origins. If the loaded app redirects, opens a malicious link, or receives untrusted content, the WebView can leave your intended surface.

Add:

* `shouldOverrideUrlLoading`
* allowed origin check against `BuildConfig.MAP_URL` / terminal base URL
* external browser handoff for off-origin URLs
* `allowFileAccess = false`
* `allowContentAccess = false`
* HTTPS-only policy
* SafeBrowsing where supported

## MUST: debug APK release workflow

The Android workflow is explicitly a debug build at `.github/workflows/android-release.yml:1-6`, runs `assembleDebug` at `67`, and publishes as `prerelease: false` at `99-106`.

That is dangerous release semantics. It should be one of:

* internal debug artifact, clearly prerelease; or
* real release APK/AAB with release build type, signing, minification policy, lint/tests, and Play-ready provenance.

Also, `continue-on-error: true` on the build step at `android-release.yml:54-67` is awkward. The later artifact check catches missing APK, but the workflow should just fail at build failure.

## SHOULD: README/docs drift

The native app README still appears to describe older slice/demo behavior in places, including stub MWA/legacy screen assumptions. The code is ahead of the docs. Update docs so reviewers/testers do not run the wrong env names or expect old behavior.

This file improved materially.

## Good

The old serious bug where `submitOrders()` sent zero target/market fields is fixed. Current mapping includes:

* `targetClanId`
* `marketToken`
* `marketAmount`
* `maxGoldIn`
* `withdrawResources`

at `packages/shared/src/adapters/IChainClient.ts:4144-4166`.

Validation for DefendBase / MarketBuy / MarketSell is also present at `IChainClient.ts:64-100`.

## MUST: remove hardcoded contract fallback

`DEFAULT_CONTRACT_ADDRESS` is still hardcoded at `IChainClient.ts:115-116`, and the real client silently uses it when `CLAN_WORLD_CONTRACT_ADDRESS` is missing at `4089-4095`.

For production, this should hard-fail. Silent address fallback is one of the fastest ways to send orders to the wrong deployment.

## MUST: remove deprecated private-key fallback

The client still falls back to `DEPLOYER_PRIVATE_KEY` if `ELDER_WALLET_KEY_PATH` is missing at `IChainClient.ts:4197-4204`.

For dev this is convenient. For release it is unsafe and violates your “no backwards compatibility / no hidden fallbacks” direction. Require the intended key path or an explicit `ALLOW_DEPLOYER_KEY_FALLBACK=true`.

# Orchestrator review

The orchestrator is not an orchestrator yet. It is a single hardcoded order submitter:

* `CLAN_ID` defaults to `1` at `apps/orchestrator/src/main.ts:5`.
* It always submits clansman `1` to `ChopWood` in Forest at `15-21`.
* It logs a narration event and exits.

That is useful as a smoke test. It should be named and treated as one.

A real orchestrator needs:

* clan ownership / authorized elder identity,
* nonce tracking,
* cooldown tracking,
* retry/backoff,
* order planning,
* chain receipt confirmation,
* event reconciliation,
* no duplicate order submission,
* per-clan policy/strategy,
* safety around season finalization/pause,
* typed action generation instead of hardcoded payloads.

# GOLD / migration note

Because the diamond uses namespaced app storage, it is not a storage-compatible upgrade of the old monolith. That is fine, but it means preserving live GOLD requires an explicit migration path.

Best path:

1. Freeze V1 at a public cutoff block/tick.
2. Fully settle all clans to cutoff.
3. Snapshot `clanId`, `owner`, `goldBalance`, and any other carried state.
4. Publish Merkle root.
5. Add one-time `GoldMigrationFacet` / `MigrationFacet`.
6. Let owners claim migrated GOLD once.
7. Remove/freeze the migration facet after the claim window.

Do not use `AdminRecoveryFacet` as an informal migration tool for production balances. It is fine for testnet repair, but live GOLD migration should be auditable and deterministic.
