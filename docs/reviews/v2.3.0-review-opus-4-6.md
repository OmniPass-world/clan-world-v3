## Executive Summary

The `dev` branch is in good shape for a hackathon codebase. The core game logic — bandit 3-tick state machine, settlement pipeline, admin recovery — is **correctly implemented** with no critical runtime bugs in the smart contracts. The end-to-end tick pipeline (heartbeat → settlement → combat → indexer → frontend) is **coherent and verified**. Diamond storage layout is clean with no collisions or upgrade hazards.

The single blocking issue is a **duplicate import in EventTicker.tsx** that will fail TypeScript compilation. Beyond that, the main concerns are code quality (4,460-line god component in WorldMap.tsx), a few missing test cases, and minor frontend cleanup items. The gas budget is well within Base Sepolia's 30M block limit under all scenarios.

## Should Fix (before next release)

### SF-1. `injectClanResources` doesn't reset `lastSettledTick`
- **Severity**: Medium
- **File**: `packages/contracts/src/diamond/lib/LibAdminRecovery.sol:85-105`
- **Confirmed by**: Agents 2, 15
- **Detail**: Injected resources can be consumed retroactively by backlogged settlement. If admin injects resources into a clan with `lastSettledTick < currentTick`, the next settlement pass applies per-tick upkeep across the unsettled window, consuming the just-injected resources. The sibling function `reviveClansman` correctly resets `lastSettledTick` when needed.
- **Fix**: Either (a) add `clan.lastSettledTick = s.world.currentTick` before injection, (b) add a `resetSettlement` parameter, or (c) document that admin should call `settleClan()` first.

### SF-2. `injectClanResources` has no upper-bound validation
- **Severity**: Medium
- **File**: `packages/contracts/src/diamond/lib/LibAdminRecovery.sol:98-103`, `ClanWorld.sol:3089-3094`
- **Confirmed by**: Agents 1, 14
- **Detail**: Owner can inject arbitrary `uint256` amounts of any resource. While access-controlled, a compromised owner key or operational mistake could break game economics irreversibly. Solidity 0.8 checked math prevents overflow, but any sub-overflow value passes silently.
- **Fix**: Add per-call caps or document as intentional for testnet-only recovery.

### SF-3. `clearLiveRoute` untracked `setTimeout`
- **Severity**: Medium
- **File**: `apps/web/src/WorldMap.tsx:2471`
- **Confirmed by**: Agents 3, 11, 15
- **Detail**: `window.setTimeout` is called without storing the ID. On unmount, the 450ms timeout can fire and call `.removeChild()`/`.destroy()` on already-destroyed Pixi objects. The `marker.route !== route` guard provides partial mitigation.
- **Fix**: Store timeout IDs in a Set on the marker or a ref; clear them in the main cleanup function.

### SF-4. `(snap as any)?.worldPaused` unsafe cast
- **Severity**: Medium
- **File**: `apps/web/src/WorldMap.tsx:2744`
- **Confirmed by**: Agents 4, 15
- **Detail**: The `as any` cast silences type checking for a speculative schema field. Currently always evaluates to `false` (dormant code). If `worldPaused` is later added with a different shape, this line would silently produce wrong results.
- **Fix**: Replace with `const worldPaused = false; // TODO(backend): plumb through schema` until the feature is implemented.

### SF-5. `computeBanditAnimPhase` exported but untested
- **Severity**: Medium
- **File**: `apps/web/src/WorldMap.tsx:879`
- **Confirmed by**: Agent 4
- **Detail**: This pure function drives all bandit animation phase selection with 5 distinct output kinds. It's explicitly designed to be testable (exported) but has no test file.
- **Fix**: Add `apps/web/src/WorldMap.banditAnim.test.ts` with happy-path tests for each phase kind.

### SF-6. `MAX_BANDIT_EAGER_SETTLE_GLOBAL_SCAN` missing from generated TypeScript constants
- **Severity**: Medium
- **File**: `packages/shared/src/generated/constants.ts`
- **Confirmed by**: Agent 13
- **Detail**: This Solidity constant (`= 100`) is a plain numeric literal that should be captured by `gen-constants.mjs`, but it's absent from the generated TypeScript file. The codegen script may not have been re-run after the constant was added.
- **Fix**: Run `pnpm gen:constants` and verify the constant appears. If not, check the regex in `scripts/gen-constants.mjs`.

### SF-7. Missing test for `reviveDeadClansmen` with nonexistent clan
- **Severity**: Medium
- **File**: `packages/contracts/test/diamond/AdminRecoveryFacet.t.sol`
- **Confirmed by**: Agent 8
- **Detail**: `reviveDeadClansmen(999)` silently succeeds (iterates empty array, no-ops), while `injectClanResources` reverts for missing clans. This behavioral inconsistency is untested.
- **Fix**: Add test confirming behavior and either add a `require` for consistency or document the silent no-op as intentional.

### SF-8. Two direct state mutations bypass `transitionBanditState` validation
- **Severity**: Medium
- **File**: `packages/contracts/src/diamond/lib/LibBanditCombat.sol:109`, `LibBanditLifecycle.sol:53`
- **Confirmed by**: Agents 2, 9
- **Detail**: The unsettled-target deferral and no-target Camped→Camped self-loop both set `bandit.state` directly without going through the validation function. Both are intentional (the validation function would reject these transitions), but the bypass is undocumented and fragile if future logic is added to `transitionBanditState`.
- **Fix**: Add inline comments explaining why the validation function is deliberately bypassed.

## Skip / False Positive

### FP-1. Diamond missing `eagerSettleActiveDefendersForBase` in combat resolution
- **Original severity**: High (Agents 2, 9)
- **Validated by**: Agent 15 — **FALSE POSITIVE**
- **Reasoning**: The diamond's `LibHeartbeat.tick()` settles ALL clans globally (lines 36-43) BEFORE calling `resolveAttackingBandits`. The monolith's per-function eager-settle call was needed because it lacked this global pre-settle. The diamond's architecture correctly simplifies this — defenders are already settled by the time combat resolution runs.

### FP-2. Snapshot-diff useEffect ordering (stale ref via `currentTickFloat()`)
- **Original severity**: Medium (Agent 3)
- **Validated by**: Agent 11 — **FALSE POSITIVE**
- **Reasoning**: React guarantees effects fire in declaration order within a render cycle. The `snapshotRef.current` update (line 1214) always executes before the snapshot-diff effect (line 1227) that reads it via `currentTickFloat()`.

### FP-3. Mixed tabs/spaces at WorldMap.tsx:3824-3829
- **Original severity**: High (Agent 4)
- **Validated by**: Agent 15 — **DOWNGRADED to Low** (cosmetic only, no runtime impact)

### FP-4. `_targetColor` property in EventTicker.tsx computed but unused
- **Original severity**: Low (Agent 4)
- **Reasoning**: Dead code but harmless. Likely forward-looking for colored attack events. Not worth a dedicated fix.

### FP-5. BanditState enum renumbering ABI-breaking for in-place upgrades
- **Original severity**: Info (Agent 12)
- **Reasoning**: Intentional for testnet hackathon. Documented with comment in `IClanWorld.sol`. No production users exist.

### FP-6. No-target advance emits `BanditStateChanged(Camped, Camped)`
- **Original severity**: Low (Agents 2, 9)
- **Reasoning**: The `BanditMoved` event is also emitted for the region change. The Convex indexer stores all events generically and doesn't have special handling that would break on same-state events.

### FP-7. `eagerSettleBanditCandidateRegion` global scan cap untested
- **Original severity**: Medium (Agent 8)
- **Reasoning**: With MAX_CLANS=12, the global cap of 100 is never reachable. The constant exists for defense-in-depth if the clan count is ever raised. Per hackathon testing rules, this is acceptable to skip.

### FP-8. `eagerSettleActiveDefendersForBase` snapshot pattern untested
- **Original severity**: Medium (Agent 8)
- **Reasoning**: The existing `test_deadTargetCleanupReleasesDefendersAndEscapesBandit` in `BanditAttackResolution.t.sol` exercises this path implicitly. The snapshot is a defensive correctness fix. Per hackathon rules, indirect coverage is sufficient.

### FP-9. Monolith redundant dead-target check
- **Original severity**: Low (Agent 2)
- **Reasoning**: Defensive coding in the monolith. The diamond version doesn't have it. Not worth changing.
